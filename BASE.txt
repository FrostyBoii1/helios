We are building a centralized browser-based operations platform to replace the company's current spreadsheet-based workflow.

The goal is not to create a multi-file static HTML app. A static HTML/CSS/JS project is only acceptable for the frontend interface or an early mock-up. The real system needs to be a proper full-stack application with a frontend, backend, database, authentication, role-based permissions, file storage, backups, and local/network hosting support.

The purpose of the platform is to move the business away from spreadsheets as the main operating system. Spreadsheets may still be used for exports or backups, but the live business workflow should happen inside this application.

The application should be accessible through a browser, ideally hosted on my NAS so staff can access it across the local network, and potentially remotely through a secure VPN or protected access method. Multiple users should be able to use the system at the same time, with live updates or reliable refresh behaviour so staff are not working from stale information.

The system needs proper user accounts. Users should have roles such as Admin, Scheduling, Approvals, Support, Sales/Admin, or similar. The role should determine what each user sees, what they can edit, and what actions they can perform.

Every job should be stored as a proper database record, not as a spreadsheet row. A job record should contain customer details, address, phone, email, salesperson, system details, install details, approval details, documents, photos, notes, reminders, support history, job status, assigned staff, important dates, and a full activity timeline.

The platform should include a dashboard for each user. The dashboard should show work relevant to that user, such as outstanding approvals, upcoming installs, jobs needing scheduling, customers needing welcome calls, post-install calls, review requests, missing MSB photos, unresolved support cases, overdue tasks, and jobs approaching deadlines.

Job scheduling should be handled through a calendar or chronological pipeline view. A new sale should be entered through a simple modal/form, saved to the database, and placed into the correct workflow stage. Jobs should be easy to schedule, reschedule, assign, filter, and update without manually moving spreadsheet rows. Changing the install date should update the job timeline and calendar placement automatically.

Each job should support clear statuses, tags, and flags. Examples include missing MSB photos, awaiting approval, ready to schedule, booked for install, post-install call required, review request required, maintenance required, customer needs support, delayed customer, and high-priority issue. These statuses should be visually clear and searchable/filterable.

The application should include task ownership and accountability. Every recurring responsibility should have an assigned owner. If Retailer PPK belongs to a specific person, the system should show that. If welcome calls belong to Renata, the system should show that. If approvals belong to Elijah, the system should show that. This removes guesswork and prevents work from being done invisibly or assumed to belong to someone else.

Tasks should be assignable to users and linked to customers or jobs. Completed tasks should be saved to a historical log. Overdue tasks should be clearly visible. Admin users should be able to see what is pending, what is completed, who completed it, and when it was completed.

Each customer/job should have a chronological timeline. Every important action should be logged, including sale creation, document uploads, approval submissions, install bookings, reschedules, support requests, notes, reminders, staff assignments, and completed tasks. The user should be able to open a customer and immediately understand the full history without searching through emails or spreadsheets.

The system needs document and photo storage. Contracts, invoices, approvals, site photos, MSB photos, job documents, and other files should be uploaded and stored against the correct job/customer. File storage can be on the NAS or another configured storage directory, but the metadata should be tracked in the database. Future scope should allow drag-and-drop document uploads and possible extraction of customer details, case numbers, approval numbers, and invoice information.

Search must be strong. Users should be able to search by customer name, address, phone number, email, case number, approval number, install date, salesperson, job status, assigned staff member, or document reference. Searching should be faster and clearer than scrolling through spreadsheets.

The platform should include reminders and internal notifications. Examples include welcome calls, post-install calls, review requests, site checks, approval follow-ups, invoice follow-ups, support follow-ups, and overdue customer communication. Notifications should go to the relevant user based on role or assignment.

There should be shared admin workspaces. If one admin completes a shared task, it should clear for other admins and move into the historical log. This prevents double handling and lets everyone see what has already been done.

The system should include reporting and analytics. It should track installs completed, jobs rescheduled, cancellations, approvals completed, support cases resolved, sales processed, average installs per week, customer delays, staff workload, and potentially profitability. Reports should be exportable monthly.

The application must include a proper login and user management system.

Users should not just open the app and all see the same thing. Each staff member needs their own account. Admin users should be able to create new user accounts, deactivate accounts, reset passwords, and assign roles.

Each user account should include basic details such as name, email/username, password, role, account status, and date created. Passwords must be stored securely using proper hashing. Passwords should never be stored in plain text.

Roles should control what the user sees and what they can do. For example, an Admin should be able to access everything, manage users, view reports, assign tasks, edit jobs, and configure the system. A Scheduling user should mainly see installs, job dates, rescheduling tools, and scheduling tasks. An Approvals user should see outstanding approvals, upcoming install deadlines, missing documents, and approval-related tasks. A Support user should see flagged customers, technical cases, support notes, and follow-ups. A Sales/Admin user should see new sales, customer onboarding, welcome calls, documents, and general admin tasks.

When a user logs in, they should land on their own dashboard. That dashboard should show what matters to their role and what has been assigned directly to them. They should not have to dig through the whole system to find their work. Their jobs, tasks, reminders, alerts, overdue items, and relevant customer updates should be immediately visible.

Admins should be able to assign tasks, customers, jobs, support cases, approval work, and scheduling responsibilities to specific users. Once assigned, that work should appear on that user's dashboard. When completed, it should be logged historically with the user who completed it and the completion time.

The goal is that when someone logs in, the system already knows who they are, what their role is, what they are responsible for, and what they need to do next.

The application must support direct access to customer files stored on the NAS.

The NAS is currently where customer folders, documents, photos, contracts, approvals, invoices, and other job files are stored. The goal is not to duplicate everything unnecessarily into the app, but to make the app act as the clean interface for accessing those files.

Each customer/job record should be able to link directly to its corresponding NAS folder. From the job detail page, users should be able to view available documents and photos associated with that customer without manually browsing through the NAS folder structure.

The backend should be able to read from a configured NAS-mounted directory or network path. File paths should be stored in the database as references connected to the correct customer/job record. The system should allow users to open, preview, download, or upload files depending on their permissions.

The app should support common file types such as PDFs, images, Word documents, Excel files, approval documents, invoices, contracts, and photos. Images and PDFs should ideally be previewable in-browser. Other documents can be opened or downloaded.

There should be permission controls around file access. Not every user should necessarily be able to delete or modify NAS files. Admin users should have full access. Other roles may only be able to view, upload, or attach files depending on their job function.

The system should avoid breaking links if folders are renamed or moved. Ideally, the app should use a consistent customer/job folder structure and store reliable references to those folders. If a NAS folder cannot be found, the app should clearly show that the file path is missing or broken rather than failing silently.

The goal is for each customer page to become the doorway into everything stored for that customer. A user should be able to open a job, see the full timeline, see the current status, and access the related NAS files from the same place without hunting through folders manually.

Core Data Model Requirements

The application should be designed around a clear relational data model.

Customers and Jobs should be treated as separate but connected entities. A customer may have one or more jobs over time, including installs, maintenance, warranty work, upgrades, support cases, or future additional work. Do not design the system as if one customer can only ever have one job.

The core tables/entities should include:

* Users
* Roles and permissions
* Customers
* Jobs
* Job statuses
* Tags and flags
* Tasks
* Activities / timeline entries
* Documents / file metadata
* Reminders / notifications
* Support cases
* Install schedule records

Jobs should belong to customers. Tasks, documents, notes, reminders, activities, support cases, and scheduling records should be linkable to the relevant customer and/or job.

The activity timeline should be append-only. Important actions should not simply overwrite previous history. When a user changes a job status, assigns a task, uploads a file, reschedules an install, completes a task, adds a note, or changes important customer/job information, the system should create a timeline/activity entry showing what changed, who changed it, and when it happened.

Data integrity is important. Use database relationships, foreign keys, constraints, and transactions where appropriate. Avoid storing important business data only as loose text if it should be structured and searchable.

The system should account for multiple users working at the same time. Use safe database update patterns, transactions for important changes, and simple versioning or optimistic locking where useful to prevent accidental overwrites.

Search should begin simple but reliable. For the first version, basic PostgreSQL search using ILIKE and structured filters is acceptable. Full-text search can be added later if needed.

NAS files should be represented in the database as metadata records, not as blobs stored directly inside the database. Store file name, file type, relative NAS path, linked customer/job, uploaded by, uploaded time, and any relevant tags. The application should validate file paths and enforce permissions before serving, previewing, downloading, uploading, or deleting files.

For live updates, start with simple refresh behaviour or periodic polling. Do not add WebSockets until the core workflow is stable.
